# Auditor Pipeline Analysis: Breaking Confirmation Bias

## The Problem
The auditor LLM is exhibiting strong confirmation bias (sycophancy), resulting in a 100% confirmation rate for 1100 posts. When an LLM is presented with a prior classification—especially one that includes persuasive reasoning (`category_reason`) and evidence (`evidence`)—it defaults to agreeing. Evaluating existing logic is cognitively "cheaper" for the model than reasoning from scratch. 

Even with a `critique` field added for Chain of Thought, the model uses it to justify the *original* decision rather than challenge it.

## Recommended Strategy: Masked Audit + Devil's Advocate

To break the anchor bias without introducing a third tie-breaker API call or doubling costs, we should combine **Masked Audit** and **Devil's Advocate Prompting**:

1.  **Masked Audit**: Remove all persuasive fields (`category_reason`, `evidence`, `dataset_references`) from the prompt input. The auditor should only see the `category_id` and `matched_rule`. 
    *   *Why?* If the auditor reads the previous LLM's reasoning, it is highly likely to agree. By hiding the reasoning, the auditor is forced to independently verify if the `category_id` applies to the text.
    *   *Rename:* Change the key from `current_classification` to `proposed_category` to subtly downgrade its authority.

2.  **Devil's Advocate Prompting**: Force the model to generate a counter-argument *before* making a decision. 
    *   *Why?* Standard Chain of Thought tends to confirm the given answer. By explicitly requesting a flaw or an alternative category first, we disrupt the confirmation pipeline.

3.  **Low-Accuracy Framing**: Add an instruction stating that the proposed categories come from a "fast, low-accuracy draft model" and that the auditor should "expect many errors."

## Why not "Blind Audit"?
A fully Blind Audit (removing the previous category entirely) effectively turns the auditor into a second classifier. If the Blind Audit disagrees with the first classifier, you have conflicting results and no way to determine which is correct without a third "tie-breaker" call (tripling API costs) or simply discarding the first classifier's work (rendering it useless). 

The Masked Audit approach gives the model a target to verify, which is faster and cheaper, but forces it to do the reasoning work itself.
